# TKT-0004 ‚Äî No visual feedback while an Apply-to-Figma operation is in flight

## Summary
Applying a brand kit to Figma (`applyToFigma()` in `src/ui/app.js`, sending a `pluginMessage` of
type `"apply"` to `figma/plugin/code.js`) shows only a transient text toast ("Applying to
Figma‚Ä¶"/"Regrouping Color Modes‚Ä¶") while the operation runs. There is no persistent visual
busy-state: the "Apply variables"/"Regroup variables" button stays clickable the whole time (a
user could click it again before `apply-done`/`apply-error` comes back, firing a second concurrent
apply), and once the toast's timeout expires there is nothing on screen indicating work is still
underway. Add a persistent, animated busy-state indicator ‚Äî proposed as a 6px inset outline on the
plugin viewport with a moving/looping gradient ‚Äî that's visible for the operation's full duration,
however long it takes, and gate the trigger button so it can't be double-fired meanwhile.

## Acceptance
- A persistent visual indicator appears the moment the `"apply"` message is sent and remains
  visible for as long as no `apply-done`/`apply-error` reply has arrived ‚Äî not a fixed-duration
  toast, an actual busy STATE.
- The "Apply variables"/"Regroup variables" trigger is disabled (or otherwise can't re-fire) while
  that state is active, closing the current double-submit gap.
- The indicator clears on either `apply-done` or `apply-error` ‚Äî the existing toast still carries
  the success/failure detail; this ticket only adds the in-flight state, it doesn't change what
  happens at completion.
- The animation is indeterminate (a looping/moving treatment, not a percentage fill) ‚Äî the apply
  path (`code.js`'s `"apply"` handler) runs its three stages (`applyBundle`, `applyFloatPlans`,
  `applyStylePlans`) with no intermediate progress pings today, so a real percentage isn't
  available without a separate, larger change to that message contract (explicitly out of scope
  here ‚Äî see Scope/Open).
- Respects the existing `data-motion="reduced"` convention (`src/ui/styles.css`) the same way
  other animated affordances in the app already do ‚Äî reduced motion gets a static/non-animated
  busy treatment, not the moving gradient.
- Scoped to the real Figma plugin embed (where `"apply"` actually fires) ‚Äî the web-app preview
  build has no Apply-to-Figma action to indicate.

## Scope / Open
- **In scope**: a busy-state flag driven by the existing `"apply"` ‚Üí `apply-done`/`apply-error`
  message lifecycle (`src/ui/app.js`'s `applyToFigma`/`onApplyDone`/`onApplyError`), a CSS
  treatment on the plugin viewport (the `<ultimate-tokens>` host element, per
  `src/ui/styles.css:74`), and disabling the trigger button for that same window.
- **Non-goal**: no new `code.js` ‚Üí UI message types, no real incremental/percentage progress
  reporting between the three apply stages ‚Äî the indicator is indeterminate by design, matching
  what's structurally available today (see Acceptance). Adding true stage-by-stage progress would
  be contract-changing and belongs in a separate, bigger ticket if ever wanted.
- **Non-goal**: no change to the existing consent gate's own copy/warning UI, and no change to
  what the completion toast says on success/error.
- **Assumption (resolved)**: the exact visual spec (6px, inset, gradient colors/speed) is a build-
  time design detail, not fixed here ‚Äî the ticket locks in "persistent, animated, indeterminate,
  motion-safe" as the behavioral contract; the precise treatment is confirmed against the app's
  existing motion/theme tokens when built.
- **Assumption (resolved)**: gating the trigger button against double-submission is included as
  part of this ticket (not a separate one) ‚Äî it's driven by the exact same busy-state flag the
  indicator needs, so building one without the other leaves the state half-communicated.
