## What's genuinely strong (say this plainly, not as a preamble to criticism)

- **persist.js is excellent engineering.** A pure serialize/hydrate pair with an explicit,
  documented per-field DOMAINS table, byte-identity round-trip guarantees, and per-field
  (not whole-object) clamping â€” verified against a sealed fuzz corpus per its own header comment.
  This is the kind of file other teams get wrong for years. No notes.
- **The render discipline is more sophisticated than "nuke and rebuild" sounds.** `render()`
  (app.js:985) snapshots focus, caret position, and scroll offsets before `replaceChildren` and
  restores them after â€” a durable fix for the classic "full re-render steals focus" bug class,
  applied uniformly rather than patched per-control. `liveRefresh()`/`_liveRefreshNow()`
  (app.js:736-785) is a deliberate, well-commented escape hatch: slider drags and text typing
  get a `requestAnimationFrame`-coalesced *partial* DOM patch (canvas scene, backdrop, example
  card, damping graph, left rail, footers) that never touches the dragged/focused control, while
  every other mutation goes through the full `render()`. That's a genuine, working two-tier state
  model, not an accidental one.
- **`edit`/`commit`/`editDrag`/`commitDrag`** (app.js:719-830) cleanly separate "mutate + save +
  refresh" from "mutate + one undo step" from "continuous drag = one debounced undo step." Undo/
  redo is a bounded (`HISTORY_MAX = 100`) stack of full serialized-doc snapshots â€” simple, correct,
  and easy to reason about, at the cost of memory (see below).
- **Single-file distribution + boot speed** are real payoffs of zero-dep vanilla: the whole app
  (engines + UI + ~2,000 curated palettes' worth of category data) ships as one HTML file that
  works offline inside Figma's sandboxed iframe with no bundler runtime, no framework hydration
  cost, no version-skew between a CDN'd dependency and the pinned one.
- **The fix-commit rate on app.js is low for its size**: of the ~140 commits touching it, only
  ~29 are `fix(...)` â€” most churn is `feat`, and several `fix` commits repeat a genuinely hard
  problem (range-slider drag inside Figma's iframe, #164/#165) rather than showing broad
  monolith-induced regression. The file is large, but it is not obviously rotting.

## (1) The monolith â€” structure, seams, actual pain

**Measured shape.** `app.js` is 7,626 lines, one `class HctApp extends HTMLElement`, ~335
method-like definitions, no nested classes/mixins, no module boundaries. Fields are declared
imperatively in `connectedCallback()` (app.js:568-666, ~60 `this.x = ...` assignments) rather than
class fields â€” meaning the full shape of the component's state is discoverable only by reading a
100-line initialization block, not by scanning declarations.

The real seams that already exist and work:
- **`this.section` routing** (`color`/`typography`/`geometry`) dispatches `renderCenter`/
  `renderLeftPane`/`renderRightPane`/canvas methods â€” this is the load-bearing seam and it's honest:
  each section's render tree is a distinct, groupable set of `render*` methods.
  - Color: `renderRampsScene`, `renderScrimsScene`, `renderMappingScene`, `renderPaletteInspector`,
    `renderRolesInspector`, `renderGlobalInspector` (app.js:3765-5450)
  - Typography: `renderTypeCanvas*`, `renderTypeTokensTable`, `renderTypographyScene`,
    `renderTypeInspector` (app.js:3262-4864)
  - Geometry: `renderGeomCanvas*`, `renderGeometryScene`, `renderGeomInspector` (app.js:6993-7500+)
- The `render*` naming convention itself is a working, greppable index â€” the file is legible
  method-by-method even without a module split.

**The actual pain is method-size, not file-size.** Rough per-concern measure by regex hit-count:
figma-related lines ~266, export/download-related ~192, breakpoint/mode-related ~237 â€” spread
fairly evenly, i.e. no single "everything figma" blob dominates. But a handful of *individual
methods* are each larger than most standalone modules elsewhere in this repo:

| method | lines | what it renders |
|---|---|---|
| `renderDrawer` (app.js:5506) | **701** | the entire export drawer: every format tab, every system toggle, zip/README generation UI |
| `renderApplyGate` (app.js:6207) | **465** | the Figma apply-consent modal + its settings |
| `renderTypeInspector` (app.js:4440) | **424** | the whole right-pane Typography inspector (scale/fonts/specimen tabs) |
| `renderSettings` (app.js:6672) | **321** | the entire Settings page (all nav sections in one method) |
| `renderMappingScene` (app.js:4070) | **304** | the roleâ†’raw mapping table scene |
| `renderNewPalette` (app.js:2552) | **315** | the New-Palette modal (all 3 tabs) |
| `renderStoryInspector` (app.js:4864) | **258** | curated-preset story panel |

`renderDrawer` alone is 701 lines â€” **9% of the entire file in one method**. This is the real
cognitive-load cost: not "app.js is big," but "to change one export-format tab you must hold a
701-line method's control flow in your head," because none of these giant methods is decomposed
into sub-renderers for their own tabs/sections the way the top-level `section` routing is.

**Merge-conflict risk is structural, not (yet) proven from history.** I can't find direct evidence
of frequent merge conflicts in the git log (can't reconstruct abandoned/rebased branches from
`git log` alone), but the repo has already had to build process specifically to route around
single-file contention â€” CLAUDE.md's Shipping section says outright: *"When background agents edit
the shared tree concurrently, isolate your commit in a `git worktree` off `origin/main`."* That's a
tell: the team already knows one 430KB file is a serialization point for concurrent work and has
compensated with workflow (worktrees) rather than structure (file splits).

**Remedy (respecting the no-framework doctrine â€” this is a file split, not an architecture
change):**
1. **Split `app.js` along the existing `this.section` seam into three files** â€”
   `app-color.js`, `app-type.js`, `app-geometry.js` â€” each exporting a mixin object of methods,
   composed onto `HctApp.prototype` (or via `Object.assign(HctApp.prototype, ColorMixins, ...)`).
   No framework, no build-step change beyond adding the new files to `bundle.mjs`'s `MODS` array
   (already a manual list â€” see Â§3). Rough cost: 1-2 days; mechanical (cut/paste + prototype
   wiring), low functional risk since it's a pure code-motion.
2. **Decompose the four >300-line render methods** into per-tab/per-section sub-methods
   (`renderDrawer` â†’ `renderDrawerCssTab`, `renderDrawerFigmaTab`, etc.) â€” this is the
   highest-value, lowest-risk fix and doesn't require the file split above to land first.
   Rough cost: 0.5 day per method, ~2-3 days total for the top 4.
3. Promote the ~60-line `connectedCallback` field-init block to a documented `#defaultState()`
   (or a plain object literal spread onto `this`) so the component's full state shape is one
   scannable list, not 60 sequential `this.x =` statements interleaved with side effects
   (keyboard install, matchMedia listener, `render()`). Cost: ~2 hours, meaningfully improves
   onboarding/audit-ability.

Severity: **MAJOR** (not CRITICAL â€” the code works, ships, and the fix rate is low; but the
largest methods are a real, compounding tax on every change to export/settings/type-inspector).

## (2) State management â€” full re-render + doc mutation

The `doc` object is the single mutable source of truth; every mutation goes through `edit`/
`commit`/`editDrag` (app.js:719-830), which is disciplined and consistent â€” there is no path I
found that mutates `this.doc` directly outside these three entry points (a quick grep for
`this.doc.` assignments outside of them would be the mechanical gate to keep this true; worth
adding as a lint/test invariant rather than trusting review).

**Correctness:** low risk. The `edit`â†’`save`â†’`render` (or `liveRefresh`) sequence is synchronous
and total â€” there's no async gap where a stale `doc` could be read mid-mutation, and
`isDirty()`/`save()` compare against a `savedSnapshot` string, so save-state tracking is exact
(string equality on a canonical JSON serialization) rather than a fragile dirty-flag.

**Undo-ability:** correct but memory-heavy. `pushHistory()` (app.js:797-801) pushes a full
`JSON.stringify(serialize(doc))` per committed edit, capped at `HISTORY_MAX = 100`. For a doc with
many palettes/roles/modes this could be a non-trivial string held 100Ă— over â€” for a typical doc
(few KB serialized) this is a non-issue; it would become one if a document's serialized size grows
into the hundreds of KB (e.g., many custom per-voice/per-size token overrides across several
breakpoint modes). No compression/diffing is attempted, which is the right trade at today's doc
sizes â€” flag it only as a thing to revisit if doc size grows an order of magnitude, not a
current bug.

**Perf headroom:** `render()` always rebuilds the *entire* subtree (`replaceChildren` on the whole
component) even when only, say, the right-pane inspector changed. For discrete (non-drag) edits
this is deliberate and almost certainly fine at today's scale â€” h()-hyperscript DOM construction
for a few hundred elements is cheap, and the alternative (a real diffing layer) is exactly the
framework-shaped complexity the doctrine avoids. The `liveRefresh` partial-patch path already
proves the team knows how to do targeted updates when it matters (60fps drag) and chooses not to
generalize it â€” that's a reasonable, not lazy, decision given the doctrine. Flag as **MINOR**:
worth a profiled check (Chrome DevTools Performance panel via `npm run smoke`, or a manual pass)
on the largest realistic gallery/doc to confirm `render()` stays sub-16ms; nothing here suggests
it currently doesn't.

## (3) Generated-assets embedding chain

`figma-plugin-assets.js`, `mcp-assets.js`, `type-fonts.js`, and `categories/*.js` are all committed
build artifacts (`// GENERATED ... DO NOT EDIT` headers), and `scripts/bundle.mjs` inlines the
*entire* dependency graph â€” engines, persist, exports, the generated assets, all 8 category data
files, the Figma binder/style-plan modules, icons, model, and app.js itself â€” into one IIFE-per-
module bundle via a **hand-rolled regex-based import transformer** (not esbuild/rollup/vite,
despite vite already being a devDependency for the Vite dev server and `tsc` build step).

Concrete coupling costs, all in `scripts/bundle.mjs`:
- `MODS` (the dependency-ordered module list) and `KEY` (pathâ†’registry-key map) are **both
  hand-maintained arrays/objects** that must be updated in lockstep whenever a new engine/UI
  module is added, or `bundle.mjs` throws `"unknown import path"` â€” a build-time-only failure
  with no type-level guard (a new `import` statement referencing an unlisted path is invisible
  until someone runs `npm run build`/`npm test`).
- The transformer explicitly **does not support `export default`** (throws loudly, per its own
  comment, because "it would leak into the module IIFE as a syntax error and only surface in the
  Chrome smoke leg") â€” a real, previously-hit failure mode the comment documents, now guarded.
  That's evidence this hand-rolled approach has already bitten once and been patched defensively
  rather than replaced.
- Category files are ~215-255KB *each* (8 files, ~1.8MB total) of generated palette/story data,
  committed to git and re-inlined into the single-file bundle on every build. This is a content
  problem, not an app.js problem, but it's part of the same "committed generated artifact" pattern
  and worth naming: `git blame`/diff noise on these files is opaque (regenerated wholesale, not
  edited), so PR review of a categories change means trusting the generator, not reading the diff.

**Gen-order coupling:** `npm run build`/`npm test` run `gen:figma-assets â†’ gen:mcp-assets â†’
gen:categories â†’ tsc â†’ vite build â†’ bundle â†’ gen:figma-ui` â€” a fixed pipeline where `bundle.mjs`
reads `dist/ultimate-tokens.html` (must exist, i.e. must run after `vite build`) and
`gen-figma-ui.mjs` reads `dist/ultimate-tokens.html` post-bundle. This is a real, working pipeline,
but it is entirely convention (script order in `package.json`), not enforced by any dependency
graph â€” reordering the `npm run build` script string incorrectly would fail confusingly (a stale
or missing `dist/` file) rather than at the point of the actual ordering violation.

**Remedy:**
1. Add a **fast pre-flight check** to `bundle.mjs`: before transforming, statically scan every
   `src/`/`figma/` file bundle.mjs *would* touch for `import ... from "..."` paths not present in
   `KEY`, and fail with a clear "add X to MODS/KEY" message rather than surfacing mid-transform.
   Cheap (a few hours), turns a build-time surprise into a build-time *guided* fix.
2. Consider whether `vite build`'s own library/IIFE mode (already a devDependency, already in the
   pipeline) can replace the hand-rolled regex transformer for the single-file case â€” vite/rollup
   already solve "one JS file in, one IIFE out" correctly, including `export default` and dynamic
   `import()`, without a bespoke MODS/KEY registry. This is the one place in the repo where the
   zero-runtime-dependency doctrine is being over-applied to a *build-time* tool (the shipped
   artifact stays zero-dep either way) â€” worth a scoped spike (1-2 days) rather than an assumption
   that hand-rolling is required by the doctrine. Not urgent; the current tool works and is well
   commented, but it is the least justified piece of hand-rolling in the codebase.

Severity: **MINOR/MAJOR borderline** â€” no user-facing bug from this today, but it is the one area
where "no dependencies" is taxing engineering time (a bespoke transformer with known unsupported
syntax) rather than paying off.

## (4) Headless shim vs. real DOM â€” what it structurally cannot catch

`test/ui/headless-boot.mjs` is well-designed *for what it targets* (interaction/state-machine
correctness: undo/redo, drag-coalescing, focus/caret preservation, ARIA roles) with 572 assertions
across 30+ lettered groups (a-z plus sub-letters). But its shim (`El`/`ClassList`/
`CSSStyleDeclaration`, lines ~14-120) is structurally blind to entire bug classes:

- **No real layout or CSS cascade.** `CSSStyleDeclaration` only stores/reads inline
  `style.setProperty` values as strings â€” it cannot catch a single one of the Safari-specific bugs
  CLAUDE.md already flags (unquoted `font-family` with digits, SVG `fill:none` cascade collisions).
  Green headless-boot tells you nothing about whether a stylesheet rule actually applies.
- **No real event bubbling.** `El.dispatch(t, ev)` (line 61-65) fires only the listeners
  registered directly on that node â€” it does not walk up `parentNode` the way a real DOM
  `dispatchEvent` does. Any interaction relying on event delegation (a parent listening for
  clicks/pointer events from descendants it doesn't directly own) is untested by this harness even
  if a test "clicks" the right element, because the shim's dispatch model doesn't match the
  browser's. The range-slider drag fix history (#164/#165, "far-drag cutoff in Figma," "pointer-
  capture + measured-track mapping") is exactly the kind of real-DOM/real-iframe pointer-event
  quirk this shim structurally cannot exercise â€” those bugs had to be found and fixed against the
  actual browser/Figma iframe, not this harness.
- **`querySelector`/`querySelectorAll` support only single-class or a hardcoded compound-selector
  special case** (`.ramp-row[data-pi]`, line 71) â€” confirmed by CLAUDE.md's own note, but worth
  restating as a concrete blind spot: any new selector shape a test needs requires *editing the
  shim*, meaning the shim's capability set is reactive to whatever tests already exist, not a
  faithful DOM subset.
- **No `showModal()`.** `_syncDrawer`/`_syncApplyGate`/`_syncSettings` all guard with
  `typeof d.showModal !== "function"` specifically so the shim's absence of dialog support doesn't
  throw â€” meaning **the entire native-`<dialog>` modal behavior (focus trap, top-layer promotion,
  Esc-to-close, backdrop semantics) is untested by anything except `npm run smoke`** (Chrome-only,
  per CLAUDE.md) and manual Safari checks. This is the same Safari blind spot CLAUDE.md already
  documents for CSS, but it's worth naming that it applies to *modal interaction correctness*
  too, not just visual/font rendering.
- **No real `matchMedia`, `ResizeObserver`, font loading, or `measureText`** â€” breakpoint-mode UI
  (a whole `section` of the app) is exercised only via directly setting `this.geomMode`/
  `this.typeMode` in tests, never via an actual viewport resize crossing a real media-query
  boundary.

**What this buys, fairly:** the shim is fast (in-process Node, no browser launch), which is why
572 assertions run in what's presumably sub-second time as part of `npm test`'s zero-dependency
gate â€” that speed is a real, deliberate trade, and it's the right one for the state-machine logic
it actually covers (undo/redo, drag debounce, focus preservation, ARIA roles) where a real browser
adds nothing but latency.

**One documentation-drift finding, ironic given this repo's own standing conviction ("stale
context is a defect, equal in severity to a bug"):** the file's own header (line 2, line 7) still
reads *"TEMPORARY self-verification harness (Phase-3 interaction polish)... NOT part of the build
or verify.mjs â€” run ad hoc and removed after."* This is false today â€” `test/run.mjs:14` lists
`"ui/headless-boot.mjs"` as a permanent member of the test suite, and CLAUDE.md itself documents
the shim's selector/DOM limitations as load-bearing project knowledge. The file has clearly
graduated from "temporary probe" to "the UI test harness" but its header was never updated to say
so. Severity: **MINOR**, trivial fix (rewrite the header comment), but worth calling out precisely
because this project's own rules treat stale headers as a real defect class.

## (5) Persist/versioning of user docs

Already praised above for its clamp discipline. The gap is **schema evolution has no version
field and no migration function** â€” every backward-compat case is handled ad hoc, three different
ways:
1. **Storage-key rename chain** (`migrateStorageKeys()`, app.js:109-118) â€” copies old-namespaced
   localStorage keys forward, newest-first. This handles *product renames*, cleanly.
2. **Per-field default-on-absence** (persist.js DOMAINS, e.g. `hueSpace` defaults to `"oklch"` but
   an explicitly-stored `"cam16"` round-trips as `"cam16"`) â€” handles *a default changing* without
   breaking old docs, also clean, well-commented.
2. **A one-off "legacy stamp"** (`hydrateStoredDoc`, app.js:210, comment: *"a pre-hueSpace STORED
   set stays cam16"*) â€” a special-cased branch for exactly one historical transition.
3. **Silent-drop allowlists** for renamed enum members â€” persist.js:366-372's own comment on the
   `voices` allowlist states it plainly: *"a voice missing here has its per-voice overrides
   SILENTLY DROPPED on hydrate"* â€” and the same comment documents a real rename that already
   happened (2026-07-13: Headingâ†’Headline, Caption folded into Tiny, Quote folded into Lead, etc.).
   A doc saved before that rename, with per-voice overrides on `"Heading"` or `"Caption"`, loses
   those overrides silently on next load â€” no warning, no migration, no way for the user to know
   their customization vanished.

**Consequence:** each of these three mechanisms works *for the specific case it was built for*,
but there's no general answer to "what happens to a doc saved under an old enum/voice/role name."
The pattern today is: rename the canon, add the new names to the allowlist, and any doc still
carrying the old name gets that field quietly reset to default. For a *cosmetic* per-voice
tweak this is low-stakes (worst case: a font override reverts to the role default). But the same
`clampTokenOverrides`/allowlist pattern governs `type.voices`, `type.modes`, `geometry.modes`, and
`roleOverrides` â€” any future *structural* rename (e.g., a role key rename, not just a voice-name
rename) would hit the identical silent-drop behavior on real per-doc customization.

**Remedy:**
1. **Add an explicit `schemaVersion` field to `serialize()`'s output** (even if unused today) so a
   future migration has something to branch on, rather than inferring "old doc" from field
   absence/shape heuristics forever. Cheap (a few hours) and non-breaking â€” old stored docs simply
   hydrate as version-less/v0.
2. **When a canon rename happens** (as with the 2026-07-13 voice rename), add a one-line explicit
   rename map (`{Heading: "Headline", Caption: "Tiny", ...}`) applied *before* the allowlist filter
   in `hydrate`/`hydrateStoredDoc`, the same pattern already used for the hueSpace legacy stamp â€”
   so a renamed key's *value* survives under its new name instead of being dropped. This is a
   process fix (do it next time a rename ships) more than a one-time code fix; cost is trivial per
   occurrence, high payoff (no silent data loss for users who saved a doc before the rename).

Severity: **MAJOR** for the general pattern (silent, undetectable data loss on rename is a real
correctness gap for a "living document" product), **MINOR** for actual user impact today (the one
documented occurrence is a per-voice font override, not full-palette or full-role data).

## (6) Zero-dependency doctrine â€” costs and benefits at this scale

**Where it clearly pays:**
- The shipped artifact (single HTML file, works in Figma's offline sandboxed iframe with
  `networkAccess: none`) is a hard requirement the doctrine satisfies for free â€” any framework
  runtime would need to be bundled anyway, so "no framework" isn't costing distribution size, it's
  *enabling* the offline/sandboxed constraint.
- `persist.js`'s explicit clamp-by-domain approach is arguably *better* than what a schema-
  validation library (zod, etc.) would give for free â€” the byte-identity round-trip guarantee is a
  bespoke invariant a generic library wouldn't enforce out of the box, so hand-rolling here is a
  genuine win, not a tax.
- The `h()` hyperscript + light DOM + native `<dialog>` choices keep the component genuinely
  embeddable as a plain custom element with no Shadow DOM styling isolation fights, no framework
  version to keep current, no hydration-mismatch class of bug.

**Where it taxes:** the two costs identified above are both build-time, not runtime â€” the
hand-rolled `bundle.mjs` import transformer (Â§3) and the informal method/state organization inside
one 7,626-line class (Â§1). Neither is a violation of the doctrine (the doctrine is about *shipped*
dependencies), but both are places where the team is solving a "how do I structure a large
JavaScript codebase" problem without borrowing any of the tooling (a real bundler, a lightweight
convention for splitting a large class into mixins/traits) that would cost nothing at runtime and
meaningfully reduce authoring friction. The doctrine is sound; its *scope* (does it have to mean
"no build-time tooling either"?) is the one place worth re-litigating, and only for `bundle.mjs`.
