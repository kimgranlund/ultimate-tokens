# TKT-0009 — One breakpoint declaration: the Typography moded collection folds into Geometry

## Summary
The Figma-variable surface declares the five breakpoint modes (Desktop · Tablet · Mobile ·
Desktop Lg · Desktop Xl) **twice** — once on the Typography collection
(`typeTokensFigmaModes`, `src/engine/type.mjs`), once on Geometry (`geomTokensFigmaModes`,
`src/engine/geometry.mjs`). Two collections means two independent mode switches per frame: a
frame can sit at Geometry=Mobile / Typography=Desktop, which is drift by construction.

Consolidate to **one moded collection — Geometry** — hosting the type variables under a
`type/` group (`type/<voice>/<step>/<prop>`), so one mode switch flips type + geometry
together. Font Primitives stays a separate single-mode collection (STRING types, no
breakpoint variance). Then migrate the live BZZR Tokens file (270 Typography variables,
141 text styles, 464 metric bindings) and delete its Typography collection.

Intake rulings (reporter, 2026-07-16, via AskUser):
1. **Grouping — `type/` prefix.** Migrated variables become `type/Display/MD/size` etc.;
   the Geometry rail stays border · focus · gap · inset · radius · size · space · type.
2. **Ticket + execute now** — engine first (otherwise the next plugin apply re-creates the
   moded Typography collection and undoes the file migration), then the BZZR file.

## Acceptance
- `typeTokensFigmaModes` emits its variables into the "Geometry" collection under `type/`;
  `_figmaFloatPlans` (src/ui/app.js) produces **one** plan for the merged collection (the
  executor prunes per-collection, so two plans on one collection would delete each other's
  variables).
- The download bundle ships one moded variables file (typography + geometry merged);
  `typography.primitives.variables.json` unchanged.

  resolve the metric pool from the Geometry collection.
- A registry-tracked "Typography" float collection left over from the two-collection era is
  retired by the executor when the merged Geometry plan (carrying `type/` variables) applies.
- BZZR Tokens file: all 270 type variables live in Geometry (5 modes, values verified
  cell-for-cell), all 141 styles bind into Geometry + Font Primitives only, the Typography
  collection is deleted, zero dangling bindings.
- `npm test` + `npm run build` green; CI green; squash-merged.

## Findings

### 2026-07-16 — built (engine + BZZR), both verified
**Engine.** `typeTokensFigmaModes` emits `type/<voice>/<step>/<prop>` into "Geometry";
`mergeModeInterchanges` (figma/binder/mode-apply-plan.mjs) folds the two halves into one
interchange — with a back-fill rule: mismatched mode lists (a configured type breakpoint beside
geometry's intrinsic set) union, and each half fills the modes it doesn't define with its own
default-mode values (one collection = one mode set; "doesn't vary there" = base values, exactly
what the two-collection era resolved). `_figmaFloatPlans` merges, validates per-half AND merged,
and attaches `retire: ["Typography"]` to a Geometry plan carrying `type/` variables; both
executors (parity-kept) retire a registry-tracked collection named in `plan.retire` — provenance
only, a user's own same-named collection survives (gated in test/figma/plugin.mjs `floatretire`).
Style planner binds `type/`-prefixed keys; the flagship's metric pool reads `byRegistry("Geometry")`.
Download bundle: `figma/tokens.modes.variables.json` replaces the typography.modes/dimension.modes
pair (zip = 62 entries). 27/27 tests green; `npm run build` clean.

**BZZR migration.** 270 vars copied into Geometry (1,350 values verified cell-for-cell, modes
mapped by name), 464 style bindings re-bound across 136 styles, node-level consumers re-pointed
(Typography page 183 + 60 nodes, Scratch 6), Typography collection deleted after a fresh
zero-consumer sweep. Final tally: 141 styles, 0 unbound, 0 dangling; Geometry = 350 vars
(80 box-geometry + 270 type/), 5 modes; spot values match the ratified tables.

**Trap found: segment-level text bindings.** ~60 mixed-styled specimen nodes ("LeBron James ·
29 PTS…") carry `fontSize`/`letterSpacing` bindings PER SEGMENT — node-level
`setBoundVariable` silently no-ops on them (no error, no effect); they need
`setRangeBoundVariable(seg.start, seg.end, field, var)` from `getStyledTextSegments(["boundVariables"])`.
`paragraphSpacing` is the inverse: NOT a substring field (throws on ranges), node-level only.
A migration must handle both layers and re-scan in the same call to prove the write took.
