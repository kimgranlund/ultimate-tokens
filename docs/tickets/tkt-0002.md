# TKT-0002 — Per-voice font granularity: the Fonts pane only exposes 5 roles, the type system has 11 voices

## Summary
The Typography editor's Fonts pane lets a user pick a font for only the 5 shared ROLES (Display,
Heading, Body, UI, Mono), but the type system names 11 distinct VOICES (Display, Heading, Sub-heading,
Kicker, Lead, Body, Quote, Caption, UI, Code, Legal) that map onto those 5 roles many-to-one (e.g.
Sub-heading and Quote both ride the `heading` role and are FORCED to share its font). Extend the font
system to per-voice granularity — any voice can carry its own family, overriding its role's shared
default — and expose that in the editor.

## Acceptance
- `typeScale`'s config gains a per-voice font-override channel (parallel to the existing
  `config.voices[voice]` weight/tracking/leading/ratio channel): a voice's RESOLVED font = its own
  override if set, else its role's font. Identity-gated — no per-voice overrides ⇒ byte-identical output
  to today.
- `persist.js`'s `clampType` round-trips the new per-voice font field (today it only recognizes the 5
  hardcoded role keys — the same "VOICES allowlist" landmine class the `type-scale` skill already warns
  about: a voice missing from an allowlist silently drops its override on hydrate).
- Every consumer that currently reads `scale.fonts[role]` resolves the per-voice override where set:
  `typeTokensCSS`, `typeTokensDTCG`'s `fontFamily` group, and `typeTokensFigmaPrimitives`'s `font/<voice>`
  alias (already a PER-VOICE alias target today — it just always points at the shared role's family
  primitive; an override should point it at a new voice-specific family primitive instead).
- The editor exposes the override — exact placement (expand the Fonts pane itself to 11 rows, vs. a
  per-voice field alongside the existing per-voice Weight/Tracking/Leading/Ratio editor in the Scale tab)
  is an open design decision, not fixed by this ticket (see Scope/Open).
- Success signal: setting Sub-heading to a font different from Heading (its shared role today) is
  reflected in the live specimen, the CSS/DTCG export, and the Figma `font/Sub-heading` alias — while a
  voice with no override stays byte-identical to current behavior.

## Scope / Open
- **UI placement** (open): expand the Fonts pane to 11 rows, or add the override to the existing
  per-voice editor panel (which already carries Weight/Tracking/Leading/Ratio/Weight-siblings per voice)
  — the existing panel's "opt-in, tuned-dot" pattern is the more consistent precedent, but this is a
  build-time design call, not fixed here.
- **Non-goal**: no new Figma export SHAPE — `font/<voice>` alias entries already exist per-voice in
  `typeTokensFigmaPrimitives`; only the resolution SOURCE needs a per-voice override, not a new
  collection/variable shape.
- **Non-goal**: no per-STEP font granularity (only per-voice) — the reported gap is 5 roles vs. 11
  voices, not voices vs. their 53 steps.
- **Non-goal**: the 5 role names/taxonomy itself is unchanged — roles stay the shared DEFAULT layer;
  voices just gain an optional escape hatch.
- **Open**: should ALL 11 voices get the override channel uniformly (simpler, consistent with how
  `config.voices` already works for weight/tracking/leading/ratio), or only the roles that currently
  SHARE across >1 voice (Display already owns its role 1:1, so a Display-level override would be a
  no-op alias)? Recommend uniform (build-time confirmation only).
