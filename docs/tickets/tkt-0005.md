# TKT-0005 — Curated presets never use the per-voice font escape hatch; give select voices their own distinct face

## Summary
Reported as "update all the presets to have the full 11 font family defined." Investigation found
no defect: all 336 curated Color Categories presets author fonts per shared ROLE only (5 slots —
`display/heading/body/ui/mono` — in each preset's `type.slots`, e.g.
`docs/reference/colors/categories/architecture.json`'s first palette), and
`design5ToTypeConfig()` (`scripts/gen-categories.mjs:167-201`) translates that into `type.fonts`
(5 keys) plus `type.voices[voice]` tuning for exactly the 5 role-primary voices
(`TYPE_VOICE_OF = {display:Display, heading:Heading, body:Body, ui:UI, mono:Kicker}`,
`gen-categories.mjs:161`) — it never writes a per-voice `.font` override. So the other 6 of 11
voices (Sub-heading & Quote → the heading role's font; Caption & Legal → the ui role's font; Lead →
the body role's font; Code → the mono role's font) always resolve via role fallback, 100%
consistently across all 336 presets. This isn't wrong — a preset's Legal voice renders in that
preset's own deliberately-chosen UI-role font (e.g. "Söhne" for the Barbican Estate preset), not a
generic default — but it predates and never adopted the per-voice font mechanism TKT-0002 (#270)
added to the engine specifically so a voice can escape its shared role font. Reclassified from a
bug report to a feature/content ticket on that basis: nothing is broken, there's just an unused
escape hatch and a coarser-than-possible result.

Clarified with the reporter: this should NOT (a) just materialize the existing fallback values as
explicit per-voice overrides (zero visual change, pure technical completeness) nor (b) author a
fully distinct font for all 6 currently-sharing voices across all 336 presets (a ~2000-decision
content undertaking). The direction is **selective differentiation**: give real, deliberately
distinct fonts to the highest-value divergent voices only, leaving the rest sharing their role's
font exactly as today.

## Acceptance
- At least one currently-sharing voice (see Scope/Open for which) gains a genuinely distinct,
  deliberately authored font across all 336 presets — not a copy of its role's font, and not a
  single hardcoded family reused everywhere (each preset's choice should still feel authored to
  that preset, consistent with how the existing 5 role fonts are curated per preset today).
- Voices NOT selected for differentiation are unchanged — still resolving via role fallback,
  exactly as today (no regression, no accidental universal materialization).
- The mechanism used is the existing engine escape hatch (`type.voices[voice].font` →
  `scale.voiceFonts` → `resolvedFontFor`, TKT-0002) — no new engine/resolution logic needed, only
  `gen-categories.mjs` (and, if the source schema needs a new field to carry the authored choice,
  `docs/reference/colors/categories/*.json`) change.
- `npm test` stays green, including `test/engine/categories.mjs` and any Figma/CSS/DTCG export
  gates that read `voiceFonts` — the newly-differentiated voices should flow through
  `--font-voice-<kebab>` (CSS), the DTCG `fontFamily` group, and the Figma `font/voice/<voice>`
  primitive alias exactly like a manually-set per-voice override does today.

## Scope / Open
- **In scope**: `scripts/gen-categories.mjs`'s `design5ToTypeConfig()`, and — if a genuinely new
  per-voice choice needs authoring rather than deriving programmatically from existing slot data —
  `docs/reference/colors/categories/*.json`'s schema (a new optional field) across some/all of the
  336 presets.
- **Non-goal**: no change to the interactive editor's Fonts tab or Scale tab (both already support
  per-voice font overrides fully, per #273/#274) — this ticket is about the CURATED CONTENT never
  using that support, not the UI that would let a user do it by hand.
- **Non-goal**: materializing the other 5 already-sharing voices as explicit-but-identical
  overrides — explicitly ruled out by the reporter (see Summary).
- **Open (design, confirm at build time)**: which voice(s) get real differentiation. Sub-heading
  and Quote both currently ride the heading role; Quote in particular is the most common
  real-world case for a deliberately distinct treatment (a pull-quote voice often reads as an
  italic or otherwise distinct serif/style against its heading, in a way Sub-heading — a
  same-register subordinate to Heading — usually doesn't need). Recommend starting with Quote
  alone and confirming whether Sub-heading, Caption, Legal, Lead, or Code earn the same treatment
  once the mechanism is proven on one voice — but this is a build-time editorial call, not fixed
  here.
- **Open (mechanical, confirm at build time)**: how the differentiated font gets chosen per
  preset — a new authored field per preset in the source JSON (`docs/reference/colors/categories/
  *.json`, most consistent with how the other 5 slots already work, but touches up to 336 entries),
  vs. a programmatic derivation/pairing rule from the existing 5 slots (less authoring effort, but
  needs a rule that still feels deliberately paired rather than mechanical — e.g. "reuse the
  heading font's family but this is exactly the kind of undifferentiated result being fixed", so a
  naive derivation risks not actually solving the problem).

## Findings

### 2026-07-17 — built: Sub-title differentiation across all 336 presets + the BZZR-shape UI ladders
Rulings (reporter, via AskUser): scope = generator change AND this ticket, voice = **Sub-title**
(the CAPS wide-tracked alternate-face heading, previously borrowing the mono font everywhere),
authoring = agent fan-out (one per category file).

**Shipped.** New optional spec field `type.faces` (voice → family) beside `slots`;
`design5ToTypeConfig` routes it through the TKT-0002 escape hatch (`voices[voice].font` →
`voiceFonts` → `resolvedFontFor`). 7 agents authored 336/336 slotted presets (48 × 7 categories;
brands' 3 slotted groups excluded — real product canons are brand decisions, not curation), each
face story-fitted, real, caps-capable, and distinct from its preset's mono/heading (display-family
width cuts allowed only where the lineage is real, e.g. Druk → Druk Wide). New `faces` gate in
`test/engine/categories.mjs` proves spec → preset → resolved-scale flow for every entry.

**Sibling change (same ask, "update the presets based on BZZR"):** the ui slot's designed weight
now also keys `UI-control` + `UI-widget` weight LADDERS (bodyClassSiblingDefaults — ladders only,
engine control-text character untouched, exactly BZZR's config shape), gated by `uiladder`.
