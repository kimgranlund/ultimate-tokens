# CTO review — ultimate-tokens core engine (src/engine/, test/engine/, docs/reference)

Scope: semantic.js, type.mjs, geometry.mjs, exports.js, derive.mjs, tonal/hct/okhsl, flags.js; the
engine test strategy (test/engine/, test/run.mjs); docs/reference as spec of record. Read-only.
`npm test` run live: **27/27 test files pass, 0 dependencies** (confirmed, not assumed).

## Verdict

The color/type/geometry math core is genuinely excellent engineering: real purity discipline, a
tested schema-evolution contract, and a parity-gate culture that has already caught and fixed a real
production incident. The one place the discipline slips is `exports.js`, which has quietly grown a
second, unrelated subsystem inside the file the color-format skill still describes as having "8
formats." Nothing here is rotten; one file needs a boundary redrawn and two sync points need a gate.

## STRONG (name it as clearly as the gaps)

1. **Purity is actually held, not just claimed.** `grep -rn "document\.|window\.|localStorage|navigator\.|querySelector" src/engine/*.js src/engine/*.mjs` → zero hits. Every engine module (hct.js, okhsl.js, tonal.js, derive.mjs, semantic.js, type.mjs, geometry.mjs) is DOM-free with an explicit "public contract" comment block naming exactly the functions the test harness imports. This is unusual discipline for a file this size (type.mjs 761 lines, exports.js 1943) and it costs nothing extra to verify — CI could add this exact grep as a standing gate for near-zero cost.
2. **The Figma-binder parity guard is a genuinely strong pattern**, not just "keep two files in sync by convention." `figma/binder/figma-semantic-binder/code.js` hand-duplicates `roleTable()` because the Figma plugin sandbox cannot `import` an ES module. `test/figma/binder.mjs:71-90` compiles the RUNTIME `code.js` source with `new Function(...)`, strips its auto-invoking `main().catch(...)` tail, calls the real `roleTable()`/`refKey()` it just loaded, and diffs the derived `{n}/{ref}` name-set against `bind-plan.mjs`'s canonical set. The comment cites the reason this exists: a real scrim-ref drift incident on 2026-06-18. This is load-bearing regression-prevention, built in response to a real bug, not aspirational tooling.
3. **`persist.js`'s clamp/allowlist model is a disciplined, explicit schema-evolution contract.** Two named invariants (roundtrip identity for in-domain fields; per-field-only clamp for out-of-domain fields) are stated up front (`src/ui/persist.js:15-27`) and every optional block (`icons`, `figmaCollections`, `export`, `voices`, `modes`, `story`, `keyColors`, …) follows one convention: attach only when non-empty/non-default, so an old config always round-trips byte-identical through a newer schema. Comments cite two specific *found-live* bugs this model was hardened against (persist.js:349-351, :392-395) — real production learning encoded as protective comments, not folklore.
4. **No dead surfaces found in the reviewed scope.** Checked every exported `exports.js` function, every `flags.js` FLAG_KEYS entry, `icon-systems.mjs`, `motion.mjs`, `ds-gates.js` for callers outside their own file — all are referenced and used by the UI, tests, or the Figma bundle. The "master enforcement switch" `TIERS_ENFORCED = false` (flags.js:22) is a deliberate pre-launch state, documented as such, not an orphaned flag.
5. **The hand-table-vs-law doctrine in type.mjs/geometry.mjs is coherent, not accreted.** Both files state explicitly, with dates and ticket IDs, which quantities are power-law-derived (icon/caret in geometry.mjs, sizes were pre-2026-07-13) and which are deliberately fixed tables that no power law fits (`CONTROL_FONT`, `SIZES`) — geometry.mjs:13-25 even says outright: "hand-authored IS the law, like type's SIZES." Retired alternatives are named inline ("retired the old v4 'caret = font' rule," geometry.mjs:16). This is unusually good self-documentation of *why* a special case exists, which is exactly what prevents special-casing from becoming cruft.

## WEAK

### 1. MAJOR — `exports.js` cohesion: the file is materially bigger than its own skill describes
**File:** `src/engine/exports.js` (1943 lines total). **Citation:** `.claude/skills/adding-export-formats/SKILL.md:18` — "The **color** formats live in `src/engine/exports.js` (8 of them)." The 8-format, auto-flow-over-roles pattern the skill documents is real and well-followed (`exportCSS`/`exportOKLCH`/`exportJSON`/`exportDTCG`/`exportUI3`/`exportTailwind` all loop `p.roles` directly; `exportShadcn` is the one documented, deliberate exception) — but that pattern only accounts for lines 1-718 (~37%). Lines 719-1930 (**1212 lines, 62% of the file**) are a completely different subsystem: `dsColorRoles`, `exportDesignSystemTokens/Spine/Components/Receipt/Bundle/StitchBundle/MakeBundle`, and a cluster of markdown/prose-template generator functions (`dsMakeButtonMd`, `dsMakeOverviewMd`, `dsMakeColorMd`, `dsMakeTypographyMd`, `dsMakeSpacingMd`, `dsMakeSetupMd`, `dsMakeGuidelinesMd` — exports.js:1560-1848) that emit DESIGN.md prose and component-preview markdown, not token serialization. This is a different *kind* of code (content authoring for the Claude Design/Google Stitch/Figma Make platform bundles) than the color-math emitters, and it is **not documented anywhere** — I grepped `docs/reference/` and the project's `.claude/skills/` for `dsColorRoles`/`exportDesignSystem*`/"Claude Design" and found only incidental mentions in `docs/marketing/fact-sheet.md` and `docs/reference/references/ui-plan.md`, neither of which is a spec of record for this code.
**Consequence:** a contributor following the file's own documented pattern (`adding-export-formats`) gets a materially incomplete model of what's in the file; a 1943-line file mixing two unrelated concerns is harder to review, diff, and blame; changes to the color-format layer and changes to the DS-bundle layer land in the same file's history with no natural separation.
**Remedy:** split into `exports.js` (the 8 color formats + shared helpers, ~740 lines — matches what the skill already describes) and a new `ds-export.js` (or `design-system-bundle.js`, ~1200 lines) for the Claude Design/Stitch/Make layer; update the skill's file list; either extend `adding-export-formats` to also own the DS layer or write a short spec of record for it under `docs/reference/`. Rough cost: half a day (mechanical extraction + import fixups across `src/ui/app.js`/`src/ui/model.mjs`/tests that import from `exports.js`, no logic changes); low risk, but check for import cycles before moving.

### 2. MINOR — manual-sync arrays without a parity gate (the class of bug the role-table gate exists to prevent, left unguarded elsewhere)
**Citation:** `src/ui/persist.js:373` hardcodes a 15-entry `VOICES` array (Display…UI-widget) that must track `type.mjs`'s `makeVoices()` keys (`src/engine/type.mjs`, the `makeVoices` function) exactly, in the same order, or a voice's per-voice override silently drops on hydrate (the comment at persist.js:368-369 says so explicitly: "a voice missing here has its per-voice overrides SILENTLY DROPPED on hydrate"). Same shape for `TYPE_TREATMENTS` (persist.js:344, 5 ids vs. `type.mjs`'s own treatment ids) and `GEOMETRY_TREATMENTS` (persist.js:432, 5 ids vs. `geometry.mjs`'s `GEOMETRY_TREATMENTS` array of objects). Unlike the semantic role table — which has an explicit, gated triple-parity (`role-table.json` ↔ `semantic.js` ↔ figma `code.js`, checked in `test/engine/semantic.mjs` + `test/figma/binder.mjs`) — there is **no test** asserting `persist.js`'s `VOICES`/`TYPE_TREATMENTS`/`GEOMETRY_TREATMENTS` literals deep-equal the engine's own lists. I confirmed by grep (`grep -n "VOICES\|TYPE_TREATMENTS\|GEOMETRY_TREATMENTS" test/ui/persist.mjs` → 0 hits).
**Consequence:** renaming, adding, or removing a voice or treatment (which the repo does routinely — TKT-0008 added UI-control/UI-widget, 2026-07-13 renamed/retired several voices) can silently corrupt or drop persisted per-voice/per-treatment customization with zero test failure. This is precisely the failure mode the 2026-06-18 scrim-drift incident taught the team to gate against for roles; the lesson hasn't generalized to these two arrays.
**Remedy:** export `VOICES`/`TYPE_TREATMENTS`/`GEOMETRY_TREATMENTS` from persist.js (or move them to a shared constant each engine file also imports) and add one assertion each in `test/ui/persist.mjs` diffing them against `Object.keys(makeVoices())` / the engines' own treatment-id lists. ~1 hour, zero design risk.

### 3. MINOR — the role-table parity guard checks less than its own name claims
**Citation:** `test/engine/semantic.mjs:114` — "`defer hpg-parity-roletable — 3-impl identity (artifact/gen.js/plugin); validated at integration`" and `test/figma/binder.mjs:78-90`, the guard this defers to. The guard diffs only the **derived name-set** (`${n}/${refKey(r.light)}` / `${n}/${refKey(r.dark)}` strings) between `semantic.js` and the Figma binder's hand-copied `roleTable()` — not the full role objects (`key`, `suffix`, ordering). Two implementations could still diverge on a role's `key` or `suffix` string while resolving to an identical ref-name-set (e.g., a typo in a variable's semantic name that happens to point at the same raw stop), and this guard would not catch it, despite its own docstring calling it "3-impl identity."
**Consequence:** narrower blast radius than the scrim-ref incident this guard was built for (which it does catch), but the claim in the comment slightly overstates the coverage — a maintainer reading "identity" might assume more than what's checked.
**Remedy:** extend the existing `loadBinder`/`roleTable` comparison in `test/figma/binder.mjs` to also diff `{key, suffix}` pairs per role (the objects are already in hand at that point), not just the derived names. ~30 min, no design change.

### 4. Opportunity, not a defect — the role table's "3 hand-kept copies + gate the diff" could become "generate 1 from 1"
`docs/reference/data/role-table.json`, `src/engine/semantic.js`'s `semanticRoles()`, and `figma/binder/figma-semantic-binder/code.js`'s `roleTable()` are three independently hand-authored implementations of the same 53-row table, kept in sync by a test-time diff rather than a single source. The Figma-sandbox constraint (no ES-module loader) is real and explicitly documented (`figma/binder/figma-semantic-binder/code.js:147-149`) — but the repo *already* has a build-time codegen/bundling pipeline (`scripts/bundle.mjs`, `scripts/gen-figma-ui.mjs`) that inlines JS for the Figma sandbox. The same technique could generate the binder's `roleTable()`/`refKey()` bodies from `semantic.js`'s real source at build time, collapsing "hand-keep-3-copies + test-gate the drift" into "generate 1 from 1, gate only that codegen ran" — eliminating the drift class instead of detecting it faster. This is a genuine payoff (removes an entire incident class) but a real one-day-plus effort with its own risk (codegen correctness, sandbox-compat verification) — flagging as a considered option, not a mandate; the current gated-copy approach is tested and has already proven it catches drift.

### 5. MINOR/architectural note — count-literal test gates are a real strength AND a real maintenance cost
`test/ui/headless-boot.mjs` is 2243 lines, 572 `ok(...)` assertions, many of them literal-count change-detectors (`=== 53`, `=== 8`, `=== 7`, `.length === 3`, `.length === 4`, etc. — e.g. lines 147, 589, 628, 632, 689, 856, 874) scattered across lettered groups, exactly as `CLAUDE.md` already flags ("keep the count literals in sync when role/step counts change"). This *is* a legitimate zero-dependency-shim design choice and it does work — a real count change will fail loudly. But the literals are unnamed magic numbers repeated at their point of use rather than named constants imported once, so a legitimate feature change (e.g., an 8th default palette) requires hunting N scattered assertions with no single place to update, and there's a real risk a future edit "fixes" an assertion to match new code rather than confirming the new count is actually correct (test tautology risk). Not asking for jsdom or a rewrite — recommend pulling the repeated magic numbers (53 roles, 25 export stops, 19 display stops, 7 scrims, 8 default palettes) into named constants at the top of the file (or importing them from `role-table.json`'s own `constants`/`rolesPerPalette` where that's literally the answer key already sitting unused by this file). ~2-3 hours, test-only, zero product risk.
