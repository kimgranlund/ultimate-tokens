# TKT-0006 — CSS and DTCG type exports only name 5 role fonts, not all 11 voices

## Summary
Reported as "update the exports to also account for the full 11 font set." Investigation found this
is real but narrower than it first sounds: the ACTUAL applied font on every one of the 53 type
tokens is already correct everywhere (CSS utility classes, DTCG's per-step composite `typography`
tokens, Figma text styles via `style-plan.mjs`, and the Figma "Font Primitives" collection all
resolve through `resolvedFontFor`, which already covers all 11 voices). Nothing renders in the
wrong font today.

What's actually incomplete: the two NAMED, standalone surfaces a consumer would reasonably reach
for to reference "voice X's font" directly (bypassing the pre-built utility classes / composite
tokens) are sparse, not dense:
- `typeTokensCSS` (`src/engine/type.mjs:379-383`) only emits a `--font-voice-<kebab>` custom
  property for voices that carry an explicit per-voice override — an un-overridden voice (e.g.
  Quote, inheriting Heading's font) has NO `--font-voice-quote` variable at all. The exported CSS
  itself establishes `--font-voice-*` as a real, discoverable naming pattern (for overridden
  voices) with no signal that it's incomplete for the rest — a consumer who reasonably assumes
  every voice has one and writes `font-family: var(--font-voice-quote)` in their own component CSS
  gets a silent wrong-font fallback (an invalid custom property, not a build error).
- `typeTokensDTCG`'s top-level `fontFamily` GROUP (`src/engine/type.mjs:439-441`) only lists 5
  role-keyed entries. A consumer scanning that group for "every font family this kit actually uses"
  sees 5, even when up to 11 distinct choices are in play via per-voice overrides.
- `typeTokensFigmaPrimitives` already does this correctly (a `font/<voice>` alias exists for all 11
  voices unconditionally, `src/engine/type.mjs:542-554`) — CSS and DTCG should match that density,
  not the other way around.

## Acceptance
- `typeTokensCSS` emits a `--font-voice-<kebab>` custom property for all 11 voices unconditionally
  (via `resolvedFontFor`), not only overridden ones. Un-overridden voices' variables simply repeat
  their role's family — same value the utility classes already apply, now also directly addressable
  by name.
- `typeTokensDTCG`'s `fontFamily` group carries one entry per voice (11), not per role (5), each
  resolved via `resolvedFontFor`. (Non-goal: no change to the per-step composite `typography`
  tokens — they're already correct.)
- Identity is NOT the bar here (unlike TKT-0002's per-voice mechanism, which had to stay
  byte-identical with no overrides set) — this is intentionally a widening of what's emitted, since
  the whole point is that every voice becomes addressable, not just overridden ones. `npm test`
  gains coverage asserting the dense 11-entry shape in both formats.

## Links
- `docs/tickets/tkt-0002.md` — the per-voice font mechanism (`resolvedFontFor`, `scale.voiceFonts`)
  this ticket makes CSS/DTCG's convenience surfaces consistent with.

## Repro
1. Open any kit with no per-voice font overrides set (the default state).
2. Export CSS (`typeTokensCSS`) — observe `--font-voice-*` never appears at all (zero voices
   overridden ⇒ zero entries), even though 11 voices exist and each resolves to a real font.
3. Export DTCG — observe the `fontFamily` token group has exactly 5 keys (`display, heading, body,
   ui, mono`), never 11.

## Expected vs actual
- Expected: both formats expose one named font-family reference per voice (11), matching
  `typeTokensFigmaPrimitives`'s existing density.
- Actual: both only expose the 5 shared roles unconditionally, plus CSS additionally exposes
  whichever voices happen to carry an explicit override — silently sparse.

## Classification
Structural / export-surface consistency — not a rendering defect (every token's actually-applied
font is correct everywhere already), but an inconsistency between export formats (Figma Primitives
is dense-for-11, CSS/DTCG are role-only-or-sparse) that risks a silent wrong-font fallback for a
consumer who reasonably assumes the documented `--font-voice-*` pattern is complete.

## Severity
minor — the primary, pre-built consumption paths (utility classes, composite DTCG tokens, Figma
text styles) are unaffected and already fully correct; this only affects a consumer directly
referencing the named convenience variables/tokens outside those paths.

## Findings
- **2026-07-12** — Root cause evident from a direct read of `src/engine/type.mjs` (no dispatch
  needed). Fixed inline: `typeTokensCSS` now emits `--font-voice-<kebab>` for all 11 voices
  unconditionally (via `resolvedFontFor`); `typeTokensDTCG`'s `fontFamily` group now has one
  voice-keyed entry per voice (11) instead of 5 role-keyed entries. `npm test` green; see the
  shipping PR for the exact diff.
